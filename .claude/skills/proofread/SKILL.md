# Proofread Skill

MarkdownファイルをAI校閲・校正するスキルです。

## 引数

```
/proofread <ファイルパス>
```

例: `/proofread drafts/article.md`

## 実行手順

以下の手順を必ず順番に行うこと。

---

### ステップ0: ファイルの読み込み

指定されたファイルを Read ツールで読み込む。
ファイルが存在しない場合はエラーを伝えて終了する。

---

### ステップ1: 校閲フェーズ

**校閲とは**: 文章の内容・構成・表現の質を審査し、改善を提案すること。

以下の観点でファイル全体を確認し、指摘事項を列挙する。

1. **表現の改善**: より明確・適切な表現に変えられる箇所
2. **説明の充実**: 補足説明があると読者の理解が深まる箇所
3. **事実確認**: 誤りや疑わしい記述
4. **内容の妥当性**: 著者の意図や記事の目的に沿わないと思われる記述
5. **構成**: 論理の流れや段落構成の問題
6. **重複・冗長**: 同じ意味・内容を繰り返している文や段落（削除または統合を提案する）
7. **表記揺れ**: 同一の固有名詞・用語が記事内で複数の書き方で使われていないか確認し、統一案を提案する

**出力形式:**

```
## 【校閲】指摘事項

1. **[行番号付近 / 該当箇所の書き出し]**
   - 指摘: （問題点の説明）
   - 提案: （具体的な修正案または改善の方向性）

2. ...
```

指摘事項がない場合はその旨を伝える。

**承認待ち**: 指摘を列挙したら、ユーザーに確認を求める。
- 全体を承認するか、個別に採否を確認するか、スキップするかを選んでもらう
- ユーザーが承認した変更のみをファイルに適用する

---

### ステップ2: 校正フェーズ

**校正とは**: 誤植・体裁上の不備を見つけて修正指示を行うこと。内容への踏み込みは行わない。

#### 2-0: 自動 lint の実行（必須）

校正フェーズを開始する前に、Bash ツールで以下を実行すること。

```
bash scripts/lint.sh <ファイルパス>
```

出力結果を確認し、検出されたすべての違反を指摘事項に含める。
HTML 属性値（`title="..."`）内のマッチは誤検知として無視してよい。
コードブロック内の行は自動除外済みのため、コードブロックの内容は指摘不要。

#### 2-1: 目視チェック

lint では検出できない以下の項目も目視で確認する。

CLAUDE.md の「校正規約」に基づき、以下を確認する。

- 日本語と英数字の間の半角スペース（句読点・括弧類が隣接する場合を除く）
- 括弧と句点の位置関係（4番の形式のみ許可）
- 半角括弧・全角括弧の使い分け
- 参考文献の表記形式
- 句読点の記事内統一
- 感嘆符・疑問符の表記
- 三点リーダの倍角使用（`……`）
- ハイフン・ダッシュの使い分け
- 脚注番号の位置（句点の前）
- 数式の記述ミス
- 強制改行の使用

**出力形式:**

```
## 【校正】指摘事項

1. **[行番号付近 / 該当箇所]**
   - 規約: （どの規約に違反しているか）
   - 修正前: `該当テキスト`
   - 修正後: `修正案`

2. ...
```

指摘事項がない場合はその旨を伝える。

**承認待ち**: 指摘を列挙したら、ユーザーに確認を求める。
- 全体を承認するか、個別に採否を確認するか、スキップするかを選んでもらう
- ユーザーが承認した変更のみをファイルに適用する

---

### ステップ3: 結果の保存と差分表示

1. 校閲・校正を経た最終版を `results/<元のファイル名>` に Write ツールで保存する
2. 変更前後の差分を Bash ツールで `diff -u <元ファイル> results/<ファイル名>` を実行して表示する
3. 完了メッセージを出力する

---

## 注意事項

- 校閲・校正とも、変更を適用する前に必ずユーザーの承認を得ること
- 校正フェーズでは内容への意見（校閲）を行わないこと
- 変更は最小限にとどめ、著者のスタイルを尊重すること
- `results/` ディレクトリが存在しない場合は Bash で `mkdir -p results` を実行してから保存する
